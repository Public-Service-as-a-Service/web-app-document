# Data contracts

TypeScript types generated from the upstream OpenAPI / Swagger specs. One sub-folder per service (`document/`, `company/`, ...). Each contains:

- `swagger.json` — the raw OpenAPI spec downloaded from the service
- `data-contracts.ts` — TypeScript interfaces/enums generated by [`swagger-typescript-api`](https://www.npmjs.com/package/swagger-typescript-api)

Our interfaces in `backend/src/interfaces/` and DTOs in `backend/src/dtos/` extend these — they are never edited by hand.

## Regenerating against the deployed gateway

When the upstream service is deployed to `api-i-test.sundsvall.se` (or whatever `API_BASE_URL` points at in `backend/.env`), run:

```bash
cd backend
yarn generate:contracts
```

This executes `src/swagger-typescript-api.ts`, which loops over the services in `src/config/api-config.ts` and for each one:

1. Curls `${API_BASE_URL}/${name}/${version}/api-docs` → `src/data-contracts/${name}/swagger.json`
2. Runs `swagger-typescript-api generate` → `data-contracts.ts`

## Regenerating against a locally running service

The `yarn generate:contracts` script builds URLs as `${API_BASE_URL}/${name}/${version}/api-docs` (e.g. `.../document/3.1/api-docs`). A locally running Spring Boot service usually exposes its OpenAPI spec at just `/api-docs` without any prefix, so the script can't reach it directly.

To generate against a local service, start it (default port `8080`) and run the two underlying steps manually from `backend/`:

```bash
# 1) Download the OpenAPI spec from the local service
curl --fail --silent --show-error --location \
  -o src/data-contracts/document/swagger.json \
  http://localhost:8080/api-docs

# 2) Generate TypeScript types
npx swagger-typescript-api generate \
  --path src/data-contracts/document/swagger.json \
  -o src/data-contracts/document \
  --modular --no-client --extract-enums
```

Swap `document` for whichever service you are regenerating, and adjust the URL/port if the service listens elsewhere.

### Verifying the spec before generating

```bash
curl -s http://localhost:8080/api-docs | python3 -m json.tool | head -40
```

Look for the expected paths/schemas to confirm the running build is the one you want to lock in.

## When to regenerate

- The upstream service adds/changes endpoints or DTOs (new fields, new request/response types)
- Before wiring frontend/backend code against new API behaviour
- After pulling upstream changes you need to integrate

Commit both `swagger.json` and `data-contracts.ts` together so the diff shows what the spec change actually was.
